绑定服务

发表于 2019-03-17

概括

（资源来自Android开发文档）

  • 绑定服务是客户端-服务器接口中的服务器。（怎么理解）绑定服务可以让组件（Activity）绑定到服务、发送请求、接受响应，甚至是执行进程间通信（IPC）
  • 绑定服务通常只在为其他应用组件提供服务时处于活动状态，不会无限期在后台运行。
  • 绑定服务的情况不止一种，因为服务的类型可以分为自身服务、来自其他应用程序的服务，本地服务、远程服务器提供的服务
  • 还有关于如何使服务设置为前台运行

Service

  • 绑定服务用到的类是Service，可以让其他应用程序与其绑定和交互。如果一个应用程序组件要绑定一个服务，需要用到onBind()回调方法，他会返回一个IBinder对象
  • IBinder实际上是一个客户端和服务之间的接口
  • 值得关注的是：我们可以将一个Service绑定到已经启动startService()的服务上，不过在服务解绑后，还需要自己调用stopSelf()或者stopService()显式停止服务。（同时实现列子：音乐播放器）
  • 客户端可以使用bindService()绑定到服务，实现时需要提供ServiceConnection的实现
  • ServiceConnection起到一个监控与服务连接的作用，一开始调用bindService的时候，马上会无值返回，但是当Android系统创建客户端和服务之间的连接时，会对ServiceConnection调用onServiceConnected()，向客户端传递用来与服务通信的IBinder
  • 多个客户端可以同时连接到同一个服务上，但是只有第一个客户端在绑定服务的时候，系统才会调用Service类中的onBind()方法来检索IBinder（思考：可以根据请求返回不同类型的binder接口？）,系统随后无需再次调用onBind(),便可将同一IBinder传递至任何其他绑定的客户端。（看到这里感觉一个服务不可以提供多种绑定接口？）
  • 当最后一个客户端解开了和该服务的绑定的时候，系统就会将该服务销毁，除非调用了startService()

IBinder

是用来提供客户端和服务之间通信的接口，在Android中提供了三种IBinder的实现方式

拓展Binder类

  • 适用情况：服务提供的是自有应用专用，并且在与客户端相同的进程中运行（常见情况），可以通过拓展Binder类并从onBind()方法中对这个实例进行返回创建接口，客户端收到Binder后，可以直接利用它直接访问Binder实现中乃至Servcie中可用的公共方法。针对同一个应用和同一个进程。

使用Messenger

  • 使用情况：当需接口跨不同的进程工作，则可使用Messenger为服务创建接口，服务可以以这种方式定义对应于不同类型Message对象的Handler。此Handler是Messenger的基础，后者（Messenger）随后可与客户端分享一个IBinder，可以让客户端利用Message对象向服务发送命令，
  • 对比线程间通信的Handler
  • 客户端可以自定以Messenger，以便回传消息
  • Message和Messenger的区别
  • 因为Messenger会在单一线程中创建包含所有请求的队列，就不需要对服务进行线程安全设计了

使用AIDL

  • AIDL执行将所有对象分解成原语的工作，操作系统可以识别这些原语并将它们编组到各进程中，以执行IPC（关注这句话的含义）。对比Messenger的方法，实际上是以AIDL作为其底层结构。如果想让你的服务同时处理多个请求，则可以使用AIDL。但是你的服务必须具备多线程处理能力，并采用线程安全设计。
  • 使用AIDL，必须创建一个定义编程接口的.aidl文件，Android SDK工具利用该文件生成一个实现接口并处理IPC的抽象类。
  • android官方表示，大多数情况下并不会使用到AIDL

拓展Binder类

  • 在服务中，创建满足以下任一要求的Binder实例：
    • 包含客户端可调用的公共方法
    • 返回当前Service实例，其中包含客户端可调用的公共方法
    • 或返回由服务承载的其他类的实例，其中包含客户端可调用的公共方法
  • 从onBind()回调方法返回此Binder实例
  • 在客户端中，从onServiceConnected()回调接收Binder，并使用提供的方法调用绑定服务
  • 需要注意：一个活动要绑定一个Service需要调用bindService，里面需要传递一个connection，里面包含了onServiceConnected()和onServiceDisconnected()，分别表示当活动绑定服务时的行为以及活动和服务断开后的行为
  • 在onServiceConnected()中包含两个参数：ComponentName和IBinder，这个函数的调用是通过android系统进行的，将绑定服务使用时onBind回调回来的Binder实例作为这个onServiceConnected函数的参数传进去，这个时候因为前面提到的，Binder里面无论是什么形式，都会包含服务中的公共方法，所以该活动就可以使用了，实现了客户端和服务的通信

使用注意

  • 在使用Binder实现服务的绑定时，若要实现在活动中可以调用服务所提供的方法，需要设置对应的onBind返回的IBinder中的内容为该活动的私有元素，如：返回的是一个Binder，就设置private ..Binder在活动（组件）中

使用Messenger

  • 与AIDL比较，在实现IPC上更加方便，前提是不考虑该服务的并发情况
  • 实际上，对于大多数应用，服务不需要执行多线程处理，所以使用Messenger就是让服务一次处理一个调用
  • 使用方法：
    • 在服务中实现一个Handler，由其来接受来自服务端的每个调用的回调
    • Handler用于创建Messenger对象，就是Messenger构造函数传进一个Handler来构建
    • 通过Messenger创建一个IBinder，服务通过onBind使其返回客户端
    • 这样看来，其实对比直接返回定义的Binder，Messenger方法相当于在Binder之上做了一层处理，然后利用getBinder方法来获得IBinder的实例
    • 在客户端，需要同样创建一个Messenger（因为他就是通信的桥梁啊），但是这个时候Messenger的构造函数不一样了，传进去的参数是，用服务端返回来的IBinder实例的引用，然后使用后者（Messenger）将Message对象发送给服务
    • 服务在其Handler中（具体地讲，是在handlerMessage()方法中）接受每一个Message
    • 这样，客户端并没有调用服务端的方法，而是将消息传递到服务上，通过Handler进行接受和处理的
    • 但是这种情况未说明服务如何对客户端做出响应，想让服务做出响应，则还需要在客户端中创建一个Messenger。然后当客户端收到onServiceConnected时候，会向向服务发送一条Message，并在其send()方法的replyTo参数中包含客户端的Messenger（属于双向消息传递）

注意：只有Activity、服务和内容提供程序可以绑定到服务，广播接受器不可以绑定服务

使用AIDL

在Android上，一个进程通常无法访问另一个进程的内存。尽管如此，进程需要将其对象分解成操作系统能够识别的原语，并将对象编组成跨越边界的对象。编写执行这一编组操作的代码是一项繁琐的工作，因此Android会使用AIDL来处理。只有允许不同应用的客户端用IPC方式访问服务（特别注意），并且想要在服务中处理多线程时，才有必要使用AIDL

  • 需要明确AIDL接口调用是直接函数调用，不应该假设发生调用的线程。（不是很懂官方文档这句话的意思）
  • 根据调用来自本地进程还是远程进程中的线程，实际情况会有所差异，具体是：
    • 来自本地进程的调用：在发起调用的同一线程内执行。如果该线程是UI线程（主线程），则该线程继续在AIDL接口中执行。如果该线程是其他线程，则其便是在服务中执行您的代码线程。因此，只有在本地线程访问服务时，才能完全控制哪些线程在服务中执行，（？？）但如果时这种情况，其实就不需要使用AIDL而是实现Binder来完成了
    • 来自远程进程的调用分派自平台在您的自有进程内部维护的线程池。您必须为来自未知线程的多次并发传入调用做好准备。换言之，AIDL接口的实现必须是完全线程安全实现。
    • onway关键字用于修改远程调用的行为。使用该关键字时，远程调用不会阻塞，它只是发送事务数据并立即返回。接口的实现最终接收此调用时，是以正常远程调用形式将其作为来自Binder线程池的常规调用进行接收。如果onway用于本地调用，则不会有任何影响，调用仍是同步调用
  • 定义AIDL接口：
    • 必须使用 Java 编程语言语法在 .aidl 文件中定义 AIDL 接口，然后将它保存在托管服务的应用以及任何其他绑定到服务的应用的源代码（src/ 目录）内（结合代码案例理解）
    • 开发每个包含 .aidl 文件的应用时，Android SDK 工具都会生成一个基于该 .aidl 文件的 .java接口文件，并将其保存在项目的 gen/ 目录中。服务必须视情况实现 IBinder 接口。然后客户端应用便可绑定到该服务，并调用 IBinder 中的方法来执行 IPC。
    • 步骤主要分为散步：
      1. 创建.aidl文件
        • 此文件定义带有方法签名的变成接口
      2. 实现接口
        • Android SDK工具基于您的.aidl文件，使用Java编程语言生成一个接口。此接口具有一个名为Stub的内部抽象类，用于拓展Binder类并实现AIDL接口中的方法。必须拓展Stub类并实现方法，本质上Stub就是Binder
      3. 向客户端公开该接口
        • 实现Service并重写onBind()以返回Stub类的实现
    • 因为其他应用如果要使用aidl定义的服务，都需要将.aidl文件复制到其他应用，才能让这些应用访问您的服务的接口。所以必须保留对原始接口的支持（尽量不改动接口）

##定义AIDL接口（根据上面步骤的具体实现）

###创建.aidl文件

// IRemoteService.aidl
package com.example.android;

// Declare any non-default types here with import statements

/** Example service interface */
interface IRemoteService {
/** Request the process ID of this service, to do evil things with it. */
int getPid();

/** Demonstrates some basic types that you can use as parameters
 * and return values in AIDL.
 */
void basicTypes(int anInt, long aLong, boolean aBoolean, float aFloat,
        double aDouble, String aString);
}
  • 这里面定义了两个接口：一个是获取当前进程id的接口，一个是指出了我需要用到作为参数的类型和作为返回值的类型（在AIDL中）
  • 只需要将这个文件保存在项目的src/目录内，当开发应用的时候，SDK工具会在项目中的gen/目录中生成IBinder接口文件（IBinder通常就是用来作为Binder的同一接口形式），生成的文件名与.aidl文件名一致，只是使用了.java拓展名（例如，IRemoteService.aidl生成的文件名是IRemoteService.java）
  • 使用android studio，可以通过增量编译马上生成这个Binder类。不使用android studio的话，Gradle工具会在下一次开发应用时生成Binder类，需要在编写完.aidl文件后马上使用gradle assembleDebug(assembleRelease)编译项目，以便代码可以链接到生成的类

实现接口

  • 有上面可以知道，我们在正确创建完.aidl文件并且放到src/文件夹下后，应用会通过编译产生对应的java接口文件（IBinder接口），生成的接口包括一个名为Stub类的子类，是对应父类（YourInterface.Stub）的抽象实现（这里android使用抽象实现是因为，一个接口如果使用abstract来实现的话，可以不完全实现接口，就可以为特定的方法留有下次实现的空间），用于声明.aidl文件中的所有方法。下面是具体的实现类：

    private final IRemoteService.Stub mBinder = new IRemoteService.Stub() {
    public int getPid(){
        return Process.myPid();
    }
    public void basicTypes(int anInt, long aLong, boolean aBoolean,
        float aFloat, double aDouble, String aString) {
        // Does nothing
    }
};
  • 上面例子中，我们可以看到很特别的一种，我们实现的不是IRemoteService的接口实现，而是调用了里面的Stub子类定义一个mBinder来进行实现（想想为什么？？）
  • 现在mBinder是Stub的一个实例，（本质上就是一个Binder），用于定义服务的RPC接口。下一步，将向客户端公开该实例，以便客户端能够和服务进行交互
  • 在实现AIDL接口时应该注意遵循以下规则：
    • 由于不能保证在主线程上执行执行传入调用，因此一开始就要做好多线程处理准备，并且将服务编译成线程安全服务（什么意思？？）
    • 默认情况下，RPC调用时同步调用，服务完成请求时间不止几毫秒，就不应该从主线程（Activity）调用服务，因为这样做可能会使应用挂起（android可能会显示ANR），所以通常做法是：从客户端内的单线程调用服务。
    • 在这个过程中引发的任何异常都不会回传给调用方

向客户端公开该接口

  • 因为实现了该接口需要公开，使得客户端可以对这个服务进行绑定
  • 公开接口，需要拓展Service（继承Service）并实现onBind(),对比刚才只是分别定义和实现了IRemoteService的接口和接口里面的一个内部类
  • 在onBind()里面实现返回了IBinder类实例（前面实现的Stub），下面代码是具体操作
public class RemoteService extends Service {
    @Override
    public void onCreate() {
        super.onCreate();
    }

    @Override
    public IBinder onBind(Intent intent) {
        // Return the interface
        return mBinder;
    }

    private final IRemoteService.Stub mBinder = new IRemoteService.Stub() {
        public int getPid(){
            return Process.myPid();
        }
        public void basicTypes(int anInt, long aLong, boolean aBoolean,
            float aFloat, double aDouble, String aString) {
            // Does nothing
        }
    };
}
  • 这个时候，当客户端调用bindService()以连接此服务的时候，客户端的onServiceConnected()回调会接受服务的onBind()方法返回IBinder实例，也就是我们在Service内部实现的IRemoteService.Stub类
  • 客户端必须具有对interface类的访问权限（有访问接口的权限），因此如果客户端和服务在不同的应用内，则客户端的应用src/目录内必须包含.aidl文件的副本（因为他会生成android.os.Binder接口-为客户端提供对AIDL方法的访问权限）
  • 当客户端在onServiceConnected()回调中收到IBinder时，因为这时候本质上就是Stub，里面包含了具体公有方法，所以可以实现客户端对服务的访问。

  • 需要注意的是，在AIDL实现中，bindService有区别，在客户端获得回调回来的IBinder实例后，还必须调用YourServiceInterface.Stub.asInterface(service)以将返回的参数转换成YourServiceInterface类型：

    IRemoteService mIRemoteService;
private ServiceConnection mConnection = new ServiceConnection() {
    // Called when the connection with the service is established
    public void onServiceConnected(ComponentName className, IBinder service) {
        // Following the example above for an AIDL interface,
        // this gets an instance of the IRemoteInterface, which we can use to call on the service
        mIRemoteService = IRemoteService.Stub.asInterface(service);
    }

    // Called when the connection with the service disconnects unexpectedly
    public void onServiceDisconnected(ComponentName className) {
        Log.e(TAG, "Service has unexpectedly disconnected");
        mIRemoteService = null;
    }
};

通过IPC传递对象

  • 通过IPC接口把一个类从一个进程发送到另一个进程是可以实现的。不过，您必须确保该类的代码对IPC通道的另一端可用，并且该类必须支持Parcelable接口。支持Parcelable接口很重要，因为Android系统可以通过他将对象分解成可编组到各进程的原语。

  • 如需创建支持Parcelable协议的类，需要进行一下操作

    1. 让定义的类实现Parcelable接口
    2. 实现writeToParcel，他会获取对象的当前状态并将其写入Parcel
    3. 为这个类添加一个名为CREATOR的静态字段，这个字段是一个实现Parcelable.Creator接口的对象
    4. 最后，创建一个声明可打包类（使用关键字parcelable来生命的类）的.aidl文件（按照下文Rect.aidl文件所示步骤）

  • 如果使用的是一个自定义的编译进程，切勿在您的编译中添加.aidl文件。此.aidl文件与c语言中的头文件类似，并未编译(暂时不理解该处？？)

  • 以下的Rect.aidl文件创建了一个可打包的Rect类，这样AIDL就可以识别到这个类（Rect），找到他对应的实现，这个很重要，不可以只实现Parcelable类

    package android.graphics;

// Declare Rect so AIDL can find it and knows that it implements
// the parcelable protocol.
parcelable Rect;

  • 通过Rect类实现了Parcelable协议

    import android.os.Parcel;
import android.os.Parcelable;

public final class Rect implements Parcelable {
    public int left;
    public int top;
    public int right;
    public int bottom;

    public static final Parcelable.Creator<Rect> CREATOR = new
Parcelable.Creator<Rect>() {
        public Rect createFromParcel(Parcel in) {
            return new Rect(in);
        }

        public Rect[] newArray(int size) {
            return new Rect[size];
        }
    };

    public Rect() {
    }

    private Rect(Parcel in) {
        readFromParcel(in);
    }

    public void writeToParcel(Parcel out) {
        out.writeInt(left);
        out.writeInt(top);
        out.writeInt(right);
        out.writeInt(bottom);
    }

    public void readFromParcel(Parcel in) {
        left = in.readInt();
        top = in.readInt();
        right = in.readInt();
        bottom = in.readInt();
    }
}

  • 对Parcel类的理解，相当于拆包后，给你定义的类里面的每一个变量贴上标签的作用，因为类本质上都是有不同的基本类型组成的。注意到这里我们定义自己的构造函数是private的，也就是说，需要调用creator来进行创建，创建的方式是读in，就是将外面的Parcel实例写进自己的类（Rect）的实例中

  • 其中Parcel的out是将自己类（Rect）写到Parcel中
  • 所以这里的in和out是相对于我们定义的类来讲的数据流方向(in:流进Rect类；out:流出Rect类到Parcel)

调用IPC方法

  • 调用类必须执行一下步骤，才能调用使用AIDL定义的远程接口：（在调用类中进行的一系列操作）

    • 在项目src/中加入.aidl文件
    • 声明一个IBinder接口实例（基于AIDL）
    • 实现ServieConnection
    • 调用Context.bindService()，以传入您的ServiceConnection实现
    • 在onServiceConnected()实现中，将会收到一个IBinder实例（名字是service），调用YourInterfaceName.Stub.asInterface((IBinder)service)，以将返回的参数转换为YourInterface类型。——-这一步很重要，这也解答了为什么在调用类里面定义的是YourInterfaceName实例（即第二步）
    • 调用在接口上定义的方法（在Stub上定义的方法），您应该始终捕获DeadObjectException异常，他们是在连接中断时才引发的；并且是远程方法引发的唯一异常
    • 断开连接使用Context.unbindService()

  • 有关调用IPC服务的几点说明：

    • 对象是跨进程计数的引用

    • 可以将匿名对象作为方法参数发送

            Toast.makeText(Binding.this, R.string.remote_service_connected,
                    Toast.LENGTH_SHORT).show();
        }

        public void onServiceDisconnected(ComponentName className) {
            // This is called when the connection with the service has been
            // unexpectedly disconnected -- that is, its process crashed.
            mService = null;
            mKillButton.setEnabled(false);
            mCallbackText.setText("Disconnected.");

            // As part of the sample, tell the user what happened.
            Toast.makeText(Binding.this, R.string.remote_service_disconnected,
                    Toast.LENGTH_SHORT).show();
        }
    };

    /**
     * Class for interacting with the secondary interface of the service.
     */
    private ServiceConnection mSecondaryConnection = new ServiceConnection() {
        public void onServiceConnected(ComponentName className,
                IBinder service) {
            // Connecting to a secondary interface is the same as any
            // other interface.
            mSecondaryService = ISecondary.Stub.asInterface(service);
            mKillButton.setEnabled(true);
        }

        public void onServiceDisconnected(ComponentName className) {
            mSecondaryService = null;
            mKillButton.setEnabled(false);
        }
    };

    private OnClickListener mBindListener = new OnClickListener() {
        public void onClick(View v) {
            // Establish a couple connections with the service, binding
            // by interface names.  This allows other applications to be
            // installed that replace the remote service by implementing
            // the same interface.
            Intent intent = new Intent(Binding.this, RemoteService.class);
            intent.setAction(IRemoteService.class.getName());
            bindService(intent, mConnection, Context.BIND_AUTO_CREATE);
            intent.setAction(ISecondary.class.getName());
            bindService(intent, mSecondaryConnection, Context.BIND_AUTO_CREATE);
            mIsBound = true;
            mCallbackText.setText("Binding.");
        }
    };

    private OnClickListener mUnbindListener = new OnClickListener() {
        public void onClick(View v) {
            if (mIsBound) {
                // If we have received the service, and hence registered with
                // it, then now is the time to unregister.
                if (mService != null) {
                    try {
                        mService.unregisterCallback(mCallback);
                    } catch (RemoteException e) {
                        // There is nothing special we need to do if the service
                        // has crashed.
                    }
                }

                // Detach our existing connection.
                unbindService(mConnection);
                unbindService(mSecondaryConnection);
                mKillButton.setEnabled(false);
                mIsBound = false;
                mCallbackText.setText("Unbinding.");
            }
        }
    };

    private OnClickListener mKillListener = new OnClickListener() {
        public void onClick(View v) {
            // To kill the process hosting our service, we need to know its
            // PID.  Conveniently our service has a call that will return
            // to us that information.
            if (mSecondaryService != null) {
                try {
                    int pid = mSecondaryService.getPid();
                    // Note that, though this API allows us to request to
                    // kill any process based on its PID, the kernel will
                    // still impose standard restrictions on which PIDs you
                    // are actually able to kill.  Typically this means only
                    // the process running your application and any additional
                    // processes created by that app as shown here; packages
                    // sharing a common UID will also be able to kill each
                    // other's processes.
                    Process.killProcess(pid);
                    mCallbackText.setText("Killed service process.");
                } catch (RemoteException ex) {
                    // Recover gracefully from the process hosting the
                    // server dying.
                    // Just for purposes of the sample, put up a notification.
                    Toast.makeText(Binding.this,
                            R.string.remote_call_failed,
                            Toast.LENGTH_SHORT).show();
                }
            }
        }
    };

    // ----------------------------------------------------------------------
    // Code showing how to deal with callbacks.
    // ----------------------------------------------------------------------

    /**
     * This implementation is used to receive callbacks from the remote
     * service.
     */
    private IRemoteServiceCallback mCallback = new IRemoteServiceCallback.Stub() {
        /**
         * This is called by the remote service regularly to tell us about
         * new values.  Note that IPC calls are dispatched through a thread
         * pool running in each process, so the code executing here will
         * NOT be running in our main thread like most other things -- so,
         * to update the UI, we need to use a Handler to hop over there.
         */
        public void valueChanged(int value) {
            mHandler.sendMessage(mHandler.obtainMessage(BUMP_MSG, value, 0));
        }
    };

    private static final int BUMP_MSG = 1;

    private Handler mHandler = new Handler() {
        @Override public void handleMessage(Message msg) {
            switch (msg.what) {
                case BUMP_MSG:
                    mCallbackText.setText("Received from service: " + msg.arg1);
                    break;
                default:
                    super.handleMessage(msg);
            }
        }

    };
}

  • 需要注意的是，里面使用到callBack显示进程调用过程中的实际情况，如果出现异常会进行捕获，因为从远程获取到callback这个过程是IPC过程，所以IPC过程都是通过线程池来分发的，也就是说不是在主线程中运行的，所以我们需要使用handler来将这个消息带回到主线程（如果需要做一下ui更新）
